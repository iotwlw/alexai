# alexai 引流入口、远程配置与高级版授权设计

## 1. 文档目的

本文档定义 alexai 作为免费引流工具发布时的页面入口、作者联系方式、远程配置和高级版授权方案。

目标包括：

- 用户可以从 Amazon 页面或扩展 Popup 找到作者、帮助与反馈入口。
- 作者可以在不重新发布扩展的情况下更新联系方式、二维码和运营文案。
- 用户可以通过授权码激活高级版。
- 作者可以远程调整授权有效期、设备数量和具体高级功能。
- 免费版保持可用，付费能力主要围绕规模、效率和持续服务展开。

本文档不包含支付系统实现，也不尝试提供无法被绕过的本地 DRM。

## 2. 核心结论

推荐采用以下三层结构：

1. 页面小图标作为轻量入口，只承载版本状态、联系作者、帮助和升级入口。
2. 扩展 Popup 或独立授权页负责展示权益、输入授权码和管理设备。
3. 远程服务负责公共配置、授权激活、续期、撤销和功能权限下发。

```text
Amazon 页面小图标
        |
        v
扩展 Popup / 授权页
        |
        v
Background Service Worker
   |                  |
   v                  v
远程配置服务        授权服务
联系方式/二维码     激活/续期/撤销
```

必须明确区分两种远程能力：

- 远程配置：更新文本、链接、二维码和功能开关，不执行远程代码。
- 远程授权：通过服务端签发的授权令牌控制正式版本中的既有功能。

核心逻辑变化仍应通过 Chrome 扩展的正式版本更新完成。禁止下载远程 JavaScript 后使用 `eval`、动态模块或其他方式执行。

## 3. 用户入口设计

### 3.1 Amazon 页面入口

当前 `content/content.js` 会在右下角显示 `alexai Ready`。建议将其改造为约 36px 的品牌图标，点击后展开轻量面板。

```text
[ alexai 图标 ]  点击
        |
        v
+----------------------+
| alexai        免费版 |
|                      |
| 使用帮助             |
| 联系作者 / 反馈问题  |
| 升级专业版           |
+----------------------+
```

交互要求：

- 默认仅显示品牌图标，避免遮挡 Amazon 商品信息。
- 悬停显示 `alexai 工具与支持` 提示。
- 点击展开面板，再次点击、点击外部或按 `Esc` 收起。
- 提供隐藏页面图标的设置，并保存用户选择。
- 使用 Shadow DOM 隔离 Amazon 页面样式。
- 授权码不直接在 Amazon 页面 DOM 中输入。
- 联系方式和升级入口由 Background 提供，不由每个页面单独联网获取。
- 不使用闪烁、强制弹窗或高频升级提醒。

### 3.2 联系作者

推荐提供以下入口：

- 小红书账号和复制按钮。
- 官方帮助页或产品落地页。
- 问题反馈入口。
- 升级专业版入口。

扩展中最好只保存一个长期稳定的落地页，例如：

```text
https://example.com/alexai
```

小红书账号、微信、公众号和二维码由该落地页或远程配置提供。这样更换渠道时无需重新发布扩展。扩展包内仍应保留一份默认联系方式作为离线兜底。

### 3.3 Popup 授权入口

建议在 Popup 顶部右侧增加版本状态按钮：

```text
alexai                              [免费版 v]
```

点击后进入账户与授权面板，展示：

- 当前版本：免费版、专业版或已过期。
- 授权码输入框和激活按钮。
- 当前授权有效期。
- 已使用和允许的设备数量。
- 刷新授权、解绑当前设备和更换授权码。
- 联系作者、隐私政策和使用条款。

必须覆盖以下状态：

- 正在激活。
- 授权码不存在或格式错误。
- 授权码已过期。
- 设备数量已达上限。
- 授权已被撤销。
- 网络不可用但仍在离线宽限期。
- 网络不可用且宽限期已结束。

## 4. 免费版与高级版边界

免费版应当可以独立解决一个完整问题，否则不利于传播和转化。高级版建议围绕处理规模、效率、自动化和持续服务收费。

| 能力 | 免费版 | 高级版 |
| --- | --- | --- |
| 页面图片和视频下载 | 保留 | 保留 |
| Alexa for Shopping/Rufus 抓取 | 每次约 20 个 ASIN | 更大批次 |
| 并发窗口 | 2 个 | 最高 5 个 |
| CSV 导出 | 保留 | 保留 |
| JSON/XLSX 和任务模板 | 可限制 | 开放 |
| 批量媒体处理 | 限制 | 开放 |
| 任务历史与断点能力 | 基础 | 完整 |
| 选择器更新和支持 | 普通 | 优先 |

以下能力不应作为付费限制：

- 合理的访问延迟和频率保护。
- 错误提示和失败保护。
- 基础隐私和数据安全能力。
- 遵守 Amazon 登录、验证码和访问限制的行为。

最自然的升级时机是用户触达免费额度时。例如用户输入 80 个 ASIN，扩展保留全部输入并说明本次处理前 20 个，升级后可以继续处理剩余任务。不要在用户尚未体验核心价值之前强制注册或频繁提示付费。

## 5. 远程公共配置

### 5.1 配置接口

Background Service Worker 定期请求只读配置接口：

```text
GET https://api.example.com/alexai/public-config
```

建议响应结构：

```json
{
  "schemaVersion": 1,
  "configVersion": 12,
  "updatedAt": "2026-07-20T12:00:00Z",
  "expiresAt": "2026-08-20T12:00:00Z",
  "contact": {
    "xiaohongshuName": "你的账号",
    "xiaohongshuUrl": "https://www.xiaohongshu.com/...",
    "supportUrl": "https://example.com/alexai/support",
    "qrUrl": "https://cdn.example.com/alexai/contact-v12.png"
  },
  "copy": {
    "upgradeTitle": "升级专业版",
    "upgradeDescription": "支持更大批次、更多导出格式和优先更新"
  },
  "featureFlags": {
    "showContact": true,
    "showUpgrade": true
  },
  "minExtensionVersion": "1.0.0",
  "signature": "base64-signature"
}
```

### 5.2 更新与缓存策略

- 扩展安装、浏览器启动和固定周期触发更新。
- 建议更新周期为 6 至 24 小时。
- 使用 `ETag` 和 `If-None-Match` 减少流量。
- 验证成功后写入 `chrome.storage.local`。
- 请求失败时继续使用最后一份有效配置。
- 没有有效缓存时使用扩展包内的默认配置。
- 二维码使用版本化 URL，避免 CDN 旧缓存。
- Background 更新配置后，通过消息或 `chrome.storage.onChanged` 通知 Popup 和 Content Script 刷新。

### 5.3 安全规则

- 配置只允许约定字段，不接受任意 HTML、CSS 和 JavaScript。
- 对文本长度、URL 协议、域名和图片尺寸进行校验。
- 外部链接只允许白名单 HTTPS 域名。
- 推荐使用 ECDSA P-256 或其他浏览器稳定支持的数字签名方案。
- 服务端保管私钥，扩展只内置公钥。
- 签名失败、配置过期或结构不合法时不得覆盖最后一份有效配置。
- 管理端写接口与扩展使用的公共只读接口必须分离。

### 5.4 二维码处理

二维码文件放在受控 CDN 或对象存储中。扩展可以通过 Background 下载并缓存，避免 Amazon 页面直接请求第三方资源。

建议限制：

- 只接受 PNG、JPEG 或 WebP。
- 限制文件大小，例如不超过 500 KB。
- 限制图片来源域名。
- 下载失败时显示默认二维码或文字联系方式。
- 二维码点击后显示实际目标说明，避免无提示跳转。

## 6. 高级版授权

### 6.1 激活流程

```text
用户输入授权码
        |
        v
Background 请求 /v1/licenses/activate
        |
        v
服务端检查授权、有效期和设备数
        |
        v
返回签名授权令牌
        |
        v
扩展验证签名并写入本地存储
        |
        v
Background 按 features 开放具体功能
```

激活请求建议包含：

```json
{
  "licenseKey": "XXXX-XXXX-XXXX-XXXX",
  "deviceId": "random-uuid",
  "extensionVersion": "1.0.0"
}
```

`deviceId` 应当由扩展随机生成并持久化，不采集硬件序列号、浏览器指纹或 Amazon 账号信息。

### 6.2 授权令牌

授权服务返回带签名的权限数据：

```json
{
  "licenseId": "lic_123",
  "plan": "pro",
  "features": [
    "large_batch",
    "json_export",
    "advanced_media"
  ],
  "issuedAt": 1784548800,
  "expiresAt": 1787227200,
  "nextCheckAt": 1784635200,
  "deviceId": "random-uuid",
  "tokenId": "token_123",
  "signature": "base64-signature"
}
```

扩展只保存签名令牌和授权状态，不保存支付信息。授权码可以在激活成功后移除，或仅保存经过处理的尾号用于展示。

### 6.3 权限判断

所有高级功能通过统一权限接口判断：

```text
hasFeature("large_batch")
hasFeature("json_export")
hasFeature("advanced_media")
```

不要在各模块散落 `plan === "pro"`。这样服务端可以调整套餐组合，而不需要修改每个功能的判断逻辑。

权限需要执行两层检查：

1. Popup 根据权限控制按钮、提示和可选配置。
2. Background 在真正执行任务前再次校验，避免只通过修改界面绕过限制。

### 6.4 续期、撤销与离线策略

- 默认每 24 小时向服务端刷新一次授权。
- 服务端可以修改功能列表、有效期和设备上限。
- 撤销授权后，下一次在线刷新应立即关闭对应高级功能。
- 网络不可用时允许 3 至 7 天离线宽限。
- 宽限期结束后退回免费版，不删除用户数据。
- 过期或撤销时保留用户输入和任务记录，避免造成数据损失。
- 可以提供仅针对高风险功能的远程关闭开关，但必须显示原因。

服务端无法保证瞬间撤销一台持续离线设备。检查周期越短，控制越及时，但服务器依赖和用户体验成本越高。

## 7. 安全与能力边界

Chrome 扩展的本地代码可以被用户查看和修改。如果高级功能完全在本地执行，授权机制只能防止普通复制，不能形成不可绕过的 DRM。代码公开在 GitHub 时，这一限制更加明显。

不应依赖以下方案保护高级版：

- JavaScript 混淆。
- 把固定授权码写入源码。
- 只在 Popup 中隐藏高级按钮。
- 把授权状态作为普通布尔值写入 `chrome.storage.local`。
- 远程下载脚本并动态执行。

如果某项能力必须获得更强保护，应将它设计成服务端能力，例如：

- 云端任务队列。
- 高级数据处理或报告生成。
- 持续更新的模板与规则库。
- 付费接口或数据服务。
- 专业版选择器更新和支持服务。

Amazon 页面抓取仍应遵守登录、验证码、访问频率、地区和权限限制。高级版不应承诺绕过这些限制。

## 8. 服务端接口与数据模型

### 8.1 建议接口

```text
GET  /alexai/public-config
POST /v1/licenses/activate
POST /v1/licenses/refresh
POST /v1/licenses/deactivate
GET  /v1/licenses/status
```

公共配置接口可以使用 CDN 缓存。授权接口不得缓存，并应具有速率限制、审计日志和防暴力尝试机制。

### 8.2 基础数据表

`licenses`：

- `id`
- `license_key_hash`
- `plan`
- `features`
- `status`
- `expires_at`
- `max_devices`
- `created_at`

`license_devices`：

- `id`
- `license_id`
- `device_id`
- `activated_at`
- `last_seen_at`
- `revoked_at`

`license_events`：

- `id`
- `license_id`
- `event_type`
- `device_id`
- `created_at`
- `metadata`

服务端只保存授权管理所需信息，不上传用户正在访问的 Amazon URL、ASIN、Cookie 或抓取结果。

### 8.3 管理后台

管理后台至少需要支持：

- 修改联系方式、帮助链接和二维码。
- 更新运营文案和入口显示状态。
- 创建、延期和撤销授权码。
- 设置套餐、功能列表、有效期和设备数量。
- 解绑指定设备。
- 查看激活、刷新、失败和撤销记录。
- 发布新配置并回滚到上一版本。

## 9. 技术选型建议

初期目标是低成本验证付费需求，不建议先开发完整账户和支付系统。

可选方案：

- Cloudflare Worker + D1 + R2：适合低成本 API、配置存储和二维码托管。
- Supabase Edge Functions + Postgres + Storage：适合需要管理后台和关系数据的场景。
- 成熟授权服务：适合已经使用其支付、订阅和税务体系的产品。

对于以小红书、微信等渠道为主的早期用户，可以将支付与授权解耦：人工收款后在后台生成授权码。确认存在稳定付费后，再接入自动支付、续费和发码。

## 10. 当前项目接入位置

建议按现有仓库边界接入：

- `manifest.json`：增加受控 API 和 CDN 域名的 `host_permissions`。
- `background/background.js`：负责远程配置拉取、签名校验、授权激活、刷新和统一权限判断。
- `content/content.js`：将 `alexai Ready` 改为品牌入口，从 Background 或 Storage 读取已验证配置。
- `popup/popup.html`：增加版本状态、联系作者和授权管理入口。
- `popup/popup.js`：处理授权 UI 状态，通过消息请求 Background 执行联网操作。
- `chrome.storage.local`：保存最后一份有效公共配置、设备 ID、授权令牌和刷新时间。

Content Script 不应持有授权服务地址、私密信息或写接口凭据，也不应在每个 Amazon 页面中重复拉取公共配置。

## 11. 分阶段实施

### 阶段一：验证引流

- 改造页面右下角品牌入口。
- 增加联系作者、帮助和升级专业版按钮。
- 建立公共配置接口和本地缓存。
- 仅统计入口点击和咨询转化，不采集 Amazon 浏览数据。

### 阶段二：人工授权 MVP

- 建立授权码数据表和激活接口。
- 使用签名令牌和统一 `hasFeature()` 判断。
- 先限制一个高价值能力，例如大批次处理。
- 人工收款、人工发码和人工解绑设备。

### 阶段三：授权运营

- 增加授权续期、撤销、设备管理和审计日志。
- 增加配置发布与回滚。
- 根据实际咨询调整免费版和高级版边界。

### 阶段四：商业自动化

- 接入自动支付和续费。
- 自动发码和到期提醒。
- 建立服务端高级能力和持续更新服务。

## 12. 验收标准

### 远程配置

- 修改联系方式或二维码后，扩展能在下一个刷新周期展示新内容。
- 配置接口不可用时仍能显示最后一份有效内容。
- 签名错误、字段非法或域名不在白名单时拒绝更新。
- 配置更新不会刷新、打断或遮挡用户当前 Amazon 操作。

### 授权功能

- 有效授权码可以激活指定功能。
- 无效、过期、撤销和设备超限状态具有明确提示。
- 修改本地普通布尔值不能直接开启高级功能。
- 授权过期或撤销后退回免费版，但不删除用户数据。
- 网络异常时按照既定宽限期工作。
- Background 与 Popup 对同一功能返回一致的授权结果。

### 隐私与安全

- 不上传 Amazon Cookie、账号、URL、ASIN 和抓取结果。
- 不执行任何远程代码。
- 授权服务私钥和管理端凭据不进入扩展源码。
- 外部链接和二维码来源均受到白名单限制。

## 13. 推荐的第一版范围

第一版只实现以下内容：

1. 页面品牌入口和联系作者面板。
2. 可远程更新的联系方式、二维码和运营文案。
3. Popup 中的免费版/专业版状态与授权码输入界面。
4. 授权激活、签名令牌、本地缓存和定期刷新。
5. 一个真实的高级功能限制，用来验证付费意愿。

暂不实现自动支付、完整用户账户、复杂订阅体系和远程代码更新。先验证用户是否点击升级、是否咨询以及是否愿意付款，再扩大后台建设范围。

## 14. 当前仓库实现映射

本仓库当前已经落地的是链接巡查专业版 MVP，使用 `D:\AmazonCode\amazon-crawler` 已有的 HTTP 接口：

- Popup 的“链接巡查”模式要求先保存授权码。
- Background 调用 `POST /api/asin-inspection`，通过 `X-Crawler-Token` 传递授权码。
- 默认服务地址为 `http://127.0.0.1:8080`；远程地址必须使用 HTTPS。
- 单次最多发送 50 条输入，Popup 展示结果并支持 CSV/JSON 导出。
- `manifest.json` 使用可选 host permissions，只有开始巡查时才申请对应服务域名权限。
- 联系方式、二维码的独立公共配置接口尚未绑定到当前扩展；目前仍使用扩展内固定文案和页面设置入口。

生产发布前必须在 `amazon-crawler` 服务端设置 `CRAWLER_API_TOKEN`。当前 API 在该环境变量为空时不会拒绝任意非空授权码；如需正式商业授权，应先补充服务端的按授权码、有效期和设备数校验，再将本扩展的令牌存储升级为签名授权令牌。
